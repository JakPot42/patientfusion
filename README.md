# PatientFusion — Fuzzy Patient Record Linkage Across Fragmented Health Silos

> **⚠ SYNTHETIC DATA ONLY.** Every patient in this project is generated by
> [Synthea](https://github.com/synthetichealth/synthea), MITRE's open-source
> synthetic patient population simulator. No real patient data is used
> anywhere in this repository. **This is a data-fusion technique
> demonstration, not a real clinical decision support tool, validated
> clinical software, or anything that should inform real patient care.**
> Every drug interaction, screening rule, and risk score below cites a real
> published source — but the *patients* they're applied to don't exist.


---

## What It Does

The US has no national patient identifier — Congress barred HHS from
funding one in the 1998 appropriations act, and the bar has been renewed
every year since. Real health systems (a primary care clinic, a retail
pharmacy, a reference lab, a hospital ED) each capture a patient's identity
independently, on paper or in incompatible EHRs, and link records the only
way they can: fuzzy name + date-of-birth matching.

PatientFusion reproduces that problem and solves it end to end:

1. **Fragment** — one real Synthea-generated population (113 patients) is
   split into four independent silos (primary care, pharmacy, labs, ER),
   each recapturing the patient's name and DOB on its own, with realistic
   per-silo formatting differences and a controlled rate of transcription
   noise (typos, day/month swaps) — see `data_prep/build_silos.py`.
2. **Link** — `entity_resolver.py` fuzzy-matches patients back together
   across silos using name similarity (token-sort/token-set, adapted
   directly from GhostTrace, project 6) gated by a DOB check, with an
   adjudicator for the ambiguous band.
3. **Decide** — a deterministic rules layer runs on the reassembled
   timeline: drug interactions, polypharmacy, overdue screenings,
   drug-specific monitoring gaps, and a simplified LACE readmission risk
   score. Every threshold cites a real source (see below) — Claude never
   sets a threshold or makes a clinical determination.
4. **Search** — natural-language population queries ("show me everyone on
   warfarin without a recent INR check"). Claude only converts the query
   into a structured filter; deterministic code executes it.

## The Genuine Reuse Story

`entity_resolver.py` is adapted directly from
[GhostTrace's](https://github.com/JakPot42/ghosttrace) `entity_resolver.py`
(project 6), which was built to deduplicate corporate ownership name
variants ("Apex Holdings LLC" vs "APEX HOLDINGS" vs "Apex Hldgs") using
token-sort and token-set fuzzy matching. The same technique maps directly
onto patient record linkage — only the gating signal changes: GhostTrace
gates auto-merge on a jurisdiction-conflict check (two same-named entities
in different jurisdictions are a classic shell pattern); PatientFusion
gates it on date of birth (a much stronger identity signal than a
corporate name ever has). Same philosophy in both: **a missed merge is
recoverable, a wrong merge is not** — so the resolver only auto-merges on
high confidence and leaves the ambiguous band to an adjudicator.

## Data Source & Provenance

- **Generator:** [Synthea](https://github.com/synthetichealth/synthea)
  (MITRE), the open-source synthetic patient population simulator used
  across health IT research (also cited in this portfolio's biosurveillance
  project, BioMon).
- **Dataset used:** the official 113-patient CSV sample export published at
  https://synthetichealth.github.io/synthea-sample-data/ — not custom-run,
  so every clinical fact (condition, medication, lab, encounter) is exactly
  what MITRE's generator produced, unmodified.
- **What PatientFusion changes:** presentation only. `data_prep/build_silos.py`
  strips Synthea's internal patient UUID (no real fragmented health system
  would share one), strips Synthea's numeric name-uniqueness suffixes
  (e.g. "Malvina930" → "Malvina"), and re-captures each patient's name/DOB
  independently per silo with realistic formatting differences and a small,
  seeded rate of transcription noise. The underlying clinical data is
  untouched.
- **Ground truth:** `data/ground_truth.csv` maps every silo record back to
  its real Synthea patient ID. The app never reads this file — it exists
  only so `tests/test_linkage.py` can measure the resolver's precision and
  recall against a known-correct answer key.

## Entity Resolution Results (measured, not claimed)

Run `patientfusion resolve` to reproduce these numbers against the seeded
demo data:

| Configuration | Master patients found | Wrong merges | Missed merges (over-splits) |
|---|---|---|---|
| No adjudicator (name+DOB gating only) | 122 | **0** | 9 of 113 |
| With adjudicator | **113 (exact)** | **0** | 0 |

**Zero wrong merges in both configurations** — the dangerous failure mode
(blending two real patients' medical histories) never happens, even with
no adjudicator at all. Without an adjudicator, 9 patients get split across
two master records instead of one (the safe failure mode: a doctor sees an
incomplete timeline, not a corrupted one). The default demo adjudicator
(`claude_adjudicator.py`, DEMO_MODE heuristic) closes that gap by requiring
first-name similarity in addition to the name+DOB signal already checked —
see "A Real Edge Case" below for why that extra check matters.

### A Real Edge Case (found in the demo dataset, not constructed)

Synthea's random generation produced two genuinely distinct patients —
**Anderson Senger** and **Shon Senger** — who share both a surname *and* an
exact birthdate (1969-05-31) by coincidence. A naive adjudicator that
merges on "name similarity + DOB match" alone wrongly merges them (verified:
an always-merge adjudicator produces exactly this false merge). The demo
adjudicator additionally checks first-name similarity, correctly keeps them
separate, and still correctly merges genuine transcription variants like
"Rocco Zulauf" / "Rocco Zualuf" (same DOB, one-letter swap). This is a real,
useful thing to discuss in an interview: fuzzy matching on any single
signal (name alone, or name+DOB alone) has a genuine failure mode, and the
fix is a cheap, explainable additional check — not a bigger model.

## Decision Layer — Cited Sources, Not Invented Thresholds

| Rule | Source |
|---|---|
| Warfarin + NSAID (ibuprofen/naproxen) | Holbrook AM et al., *Arch Intern Med* 2005;165(10):1095-1106 |
| Warfarin + Aspirin | Holbrook AM et al., *Arch Intern Med* 2005;165(10):1095-1106 |
| Digoxin + Verapamil | Lanoxin (digoxin) FDA Prescribing Information, Drug Interactions |
| Simvastatin + Amlodipine (20mg/day cap) | FDA Drug Safety Communication, June 8, 2011 |
| Opioid + Benzodiazepine | FDA Drug Safety Communication, Aug 31, 2016 (boxed warning) |
| ACE inhibitor + ARB (dual RAAS blockade) | Mann JF et al. (ONTARGET), *Lancet* 2008;372(9638):547-553 |
| Polypharmacy (5+ concurrent meds) | Masnoon N et al., *BMC Geriatr* 2017;17:230 |
| Diabetes screening (A1c, age 35-70) | USPSTF (2021), Grade B |
| Lipid panel (every 4-6 years, age 20+) | 2018 AHA/ACC Multi-Society Blood Cholesterol Guideline |
| Warfarin → INR monitoring (30 days) | CHEST 2012 Antithrombotic Therapy Guidelines |
| Metformin → renal function (annual) | ADA Standards of Care in Diabetes |
| Readmission risk (LACE index) | van Walraven C et al., *CMAJ* 2010;182(6):551-557 |

Full text of every rule/citation is in `config.py` — nothing above is
paraphrased from memory; each is a real, checkable source.

**Disclosed simplifications** (not hidden):
- The USPSTF A1c screening criterion also requires overweight/obesity
  status; this demo applies the age band alone (BMI isn't in the dataset's
  scope — see below).
- The LACE "Comorbidity" component here is a presence-count over a curated
  Charlson-category keyword list, not the full weighted Charlson
  Comorbidity Index.

## Known Data Limitations (disclosed, not hidden)

- **Synthea's public module set does not generate INR/coagulation lab
  panels**, even for patients on warfarin. The "warfarin without a recent
  INR check" query is fully implemented and generically correct — it
  works on any drug/lab pair — but in this specific demo dataset every
  warfarin patient will show as missing an INR record, because the source
  data has none. Run `patientfusion demo` to see this called out directly
  in the output, alongside a second natural-language query (metformin /
  eGFR) that Synthea's data does support, with genuine positive and
  negative results.
- Vital signs (blood pressure, BMI, height/weight) were excluded from the
  `labs` silo to keep it focused on lab-panel results (Synthea tags them
  as a separate `vital-signs` category); age-based screening rules that
  would normally also check BMI are documented above where that applies.

## Architecture

```
config.py               All thresholds, cited drug interactions, screening rules — no logic
data_prep/build_silos.py  One-time: raw Synthea export -> four fragmented silo CSVs + ground truth
silo_loader.py           Reads a silo CSV, groups rows by (name, DOB) identity
entity_resolver.py       Fuzzy name+DOB patient matching (adapted from GhostTrace)
claude_adjudicator.py    Adjudicates the ambiguous match band (DEMO_MODE heuristic or live Claude)
linkage.py               Orchestrates resolution across all 4 silos -> MasterPatient objects
models.py                MasterPatient dataclass
dates.py                 Shared date parsing for clinical fields
drug_interactions.py     Deterministic drug-drug interaction checker (cited rules)
polypharmacy.py          5+ concurrent medications (cited threshold)
screenings.py            Overdue population screenings + drug monitoring gaps (cited rules)
readmission_risk.py      Simplified LACE readmission risk score (cited, real instrument)
risk_engine.py           Aggregates all of the above per patient / across the population
nlsearch.py              Natural-language search: Claude extracts a filter, code executes it
dashboard.py             Rich terminal rendering (ASCII-safe)
main.py                  Click CLI
```

## Quick Start

```bash
git clone https://github.com/JakPot42/patientfusion.git
cd patientfusion
python -m venv venv
venv\Scripts\pip install -r requirements.txt
venv\Scripts\python main.py demo
```

The silo data (`data/silos/*.csv`) is already built and checked into the
repo — no download or Synthea installation needed to run the app. To
regenerate it from scratch: download the sample export from
https://synthetichealth.github.io/synthea-sample-data/downloads/latest/synthea_sample_data_csv_latest.zip,
unzip `patients.csv`, `encounters.csv`, `conditions.csv`, `medications.csv`,
and `observations.csv` into `data/raw/`, then run
`python data_prep/build_silos.py`.

## Commands

```bash
python main.py resolve                 # entity resolution stats
python main.py list                    # list resolved master patients
python main.py timeline MP-00113       # one patient's cross-silo timeline + findings
python main.py heatmap --top 20        # population risk heatmap
python main.py search "..."            # natural-language population search
python main.py demo                    # everything, end to end
```

DEMO_MODE=True (default) uses a deterministic keyword parser and adjudicator
heuristic — zero API keys required, exactly like every other project in
this portfolio. Set `DEMO_MODE=False` with a valid `ANTHROPIC_API_KEY` to
use live Claude calls for search-query parsing and ambiguous-match
adjudication (Claude still never sets a score or makes a clinical call —
see "Claude extracts, rules decide" above).

## Honest Limitations

- 113 patients is a small population — real fragmented health systems
  operate at a scale where fuzzy matching produces far more ambiguous
  pairs; this demonstrates the technique, not production-scale throughput.
- The demo adjudicator's first-name-similarity heuristic is a reasonable
  default, not a substitute for a live Claude call (or a human) reviewing
  full context (address, gender, clinical plausibility) on every
  ambiguous pair in a real deployment.
- LACE's Comorbidity component is simplified (see above) — a real
  implementation would use the full weighted Charlson Comorbidity Index.
- No authentication, no encryption at rest — this is a CLI demo, not a
  HIPAA-eligible system.
